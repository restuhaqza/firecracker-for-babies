## üìã Contribution Guidelines

### Before You Start

1. **Read the existing content** - Familiarize yourself with what's already there
2. **Check open issues** - Someone might already be working on your idea
3. **Think about the audience** - Keep it beginner-friendly!

### Making Changes

1. **Fork the repository**
   ```bash
   # Click the "Fork" button on GitHub
   ```

2. **Clone your fork**
   ```bash
   git clone https://github.com/YOUR_USERNAME/firecracker-for-babies.git
   cd firecracker-for-babies
   ```

3. **Create a branch**
   ```bash
   git checkout -b feature/your-feature-name
   # or
   git checkout -b fix/your-bug-fix
   ```

4. **Make your changes**
   - Edit the Markdown files
   - Follow the existing style
   - Keep explanations simple and clear
   - Use analogies where appropriate

5. **Commit your changes**
   ```bash
   git add .
   git commit -m "Brief description of changes"
   ```

6. **Push to your fork**
   ```bash
   git push origin feature/your-feature-name
   ```

7. **Create a Pull Request**
   - Go to the original repository
   - Click "New Pull Request"
   - Describe your changes clearly

### Style Guidelines

#### Tone and Voice
- **Friendly and encouraging**: Make readers feel capable
- **Simple language**: Avoid jargon or explain it
- **Conversational**: Write like you're explaining to a friend
- **Empathetic**: Acknowledge that some concepts are hard

#### Formatting
- Use Markdown for everything
- Use tables for comparisons
- Use code blocks for commands
- Use emojis for visual interest (but don't overuse)
- Keep lines under 80 characters when practical

#### Structure
- Start with a simple overview
- Use analogies before technical details
- Provide step-by-step instructions
- Include examples for every concept
- End with a summary or quiz

#### What We Love to See:
- Real-world analogies
- Step-by-step tutorials
- Common mistakes to avoid
- Practice exercises
- Links to additional resources

#### What to Avoid:
- Overly technical jargon
- Assumptions about prior knowledge
- Overly detailed information (keep it focused)
- Walls of text (break it up)
- Overuse of emojis

## üêõ Reporting Issues

When reporting issues, include:

1. **What you were trying to do**
2. **What you expected to happen**
3. **What actually happened**
4. **Steps to reproduce**
5. **Your environment** (OS, Firecracker version, etc.)

### Issue Template

```markdown
## Issue Type
[ ] Documentation error
[ ] Unclear explanation
[ ] Missing information
[ ] Suggestion
[ ] Other

## Description
Brief description of the issue

## Location
Which file/section has the issue?
(e.g., docs/05-quick-start.md, Step 3)

## Problem
What's wrong?
Be as specific as possible.

## Suggested Fix
How should it be fixed?
(If you have an idea)

## Additional Context
Any other relevant information
```

## üé® Adding New Content

### New Chapters

If you want to add a new chapter:

1. **Check if it fits** - Does it align with the "for babies" philosophy?
2. **Create an outline** - What will this chapter cover?
3. **Open an issue** - Discuss it first before writing
4. **Follow the pattern** - Match the style of existing chapters

### New Examples

When adding examples:

1. **Make them complete** - Don't skip steps
2. **Test them** - Make sure they actually work
3. **Explain them** - Tell readers what's happening
4. **Use real scenarios** - Practical examples over theoretical ones
